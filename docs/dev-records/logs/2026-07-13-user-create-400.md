# 开发记录：user-create-400

## 完成内容

本次定位并修复了用户管理页面创建用户时只显示 `400 Bad Request`、无法知道具体原因的问题。

线上复现确认：使用密码 `123456` 创建用户时，后端返回 `Password must be at least 8 characters`。当前安全策略要求密码至少 8 位，并同时包含字母和数字。前端 API 错误解析却优先读取响应中的通用 `error: "Bad Request"`，覆盖了具体 `message`，导致页面只显示模糊错误。

修复后，创建用户和重置密码表单会在请求前校验密码，并显示中文规则；API 层会优先显示具体 `message`，用户名重复、角色不存在等错误也能正确展示。

## 实际改动文件

* `b_k3ewYvsOEc1/src/lib/apiError.js`：新增 API 错误信息解析，过滤 `Bad Request` 等通用 HTTP 文案，优先返回具体 message/details。
* `b_k3ewYvsOEc1/src/lib/userValidation.js`：新增用户密码规则校验及中文提示。
* `b_k3ewYvsOEc1/src/lib/api.js`：普通请求、刷新请求和 blob 请求统一使用具体错误解析。
* `b_k3ewYvsOEc1/src/components/UserManagement.vue`：创建用户和重置密码提交前校验；密码输入框增加 minlength、pattern、title 和可见规则说明。
* `tests/user-create-validation.test.ts`：覆盖具体后端 message、用户名重复、message 数组及密码规则。

## 关键行为决策

* 保留后端既有安全策略，不允许恢复使用纯数字 `123456` 等弱密码。
* 前端校验用于即时反馈，后端校验继续作为最终安全边界。
* API 错误解析在公共请求层修复，使其他页面也能看到后端具体错误，而不是只看到 `Bad Request`。
* 线上复现使用短密码，在数据库写入前即被拒绝，没有创建测试用户或产生脏数据。

## 后续排查入口

如果后续仍无法创建用户，优先检查：

1. 浏览器 Network 中 `/api/users` 的响应 body，而不是只看状态码。
2. 密码是否至少 8 位并同时包含字母和数字，例如 `user1234`。
3. 用户名是否已存在。
4. 提交的 roles 是否都存在于 `roles` 表。
5. 当前账号是否具有 `user:manage`。
6. 前端是否已部署包含 `apiError.js` 和 `userValidation.js` 的版本。
