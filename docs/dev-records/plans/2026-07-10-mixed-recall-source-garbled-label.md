# 开发计划：mixed-recall-source-garbled-label

## 背景

编报结果页的“混合召回”来源信息中出现 `?????(?)` 这类乱码/占位符，用户看到后无法判断来源类型或媒体名称，影响信源可信度判断。

## 本次目标

定位 `?????(?)` 是后端数据清洗、编码转换、还是前端展示兜底导致的；修复显示逻辑，避免无意义问号串出现在来源标签里，优先显示可读的中文或英文来源信息。

## 本次不做

* 不改报告生成主链路。
* 不改 RBAC、登录、数据库结构。
* 不改资料采集执行逻辑。
* 不在当前编报任务完成前提交或部署。

## 涉及范围

* 前端页面：报告详情/信源列表展示组件。
* 后端接口：如确认来源数据在接口侧已经乱码，再做最小清洗。
* 数据模型：不涉及结构变更。
* 测试文件：根据定位结果补充前端或后端格式化测试。

## 开发步骤

1. 搜索前端“混合召回”和来源信息渲染代码。
2. 搜索后端 source/media/publisher/sourceType 等字段映射逻辑。
3. 判断 `?????(?)` 是原始数据、接口响应还是前端兜底拼接产生。
4. 做最小修复：过滤纯问号乱码字段，回退到更可靠字段或“未知来源”。
5. 补充测试覆盖问号乱码来源不会直接展示。
6. 运行构建/测试验证。
7. 等用户当前编报任务完成后，再按用户要求提交和部署。

## 验证方式

* 搜索并检查相关渲染代码。
* 运行相关单元测试。
* 运行前端构建：`npx pnpm@9.15.9 --dir b_k3ewYvsOEc1 build`。
* 如修改后端，运行：`npx pnpm@9.15.9 build`。
* 页面手动验证：报告详情页“混合召回”来源不再展示 `?????(?)`。
