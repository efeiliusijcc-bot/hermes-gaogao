# 拟稿助手聊天式简化改版设计

## 1. 范围

本次只修改前端，不新增或修改数据库、NestJS 控制器、服务和接口。

目标是把拟稿助手收敛为“输入一次材料，系统自动分析，用户生成并修改提纲，确认后直接进入深度编报”的五阶段流程。视觉和历史记录交互参考当前 ChatGPT 页面，但不复制其品牌、文案、图标资产或私有实现。

本次不实现以下后端能力：

- 真正删除历史编报，因为现有接口没有删除事件的能力。
- 真正移除提纲历史版本，因为现有保存和 AI 修改接口仍会创建新版本。
- 持久化独立的 `confirm`、`importing` 阶段，因为现有事件记录没有对应字段。

前端只展示每个事件的最新提纲，不暴露版本概念。历史删除入口不制作不可用的假交互。

## 2. 设计原则

1. 首屏只有一个任务：输入编报主体和材料。
2. 页面不使用营销式布局，不堆叠装饰卡片。
3. 历史记录是辅助导航，第二至第五阶段默认收起。
4. 阅读状态采用对话式纵向信息流，编辑状态优先保证宽度和效率。
5. 用户修改始终围绕“当前最新提纲”，不出现 V1、V2、恢复版本或版本比较。
6. 沿用平台顶部导航、字体和基础颜色，使用现有图标库或 Lucide 图标。

## 3. 页面结构

### 3.1 第一阶段：输入编报主体

首屏不显示步骤条、历史列表、事件分类、地区、独立链接、补充材料、完整度、建议和预览。

主内容在可用空间中水平居中，包含：

- 标题“开始一次新的编报”。
- 一段简短占位提示，不描述功能清单。
- 一个可自动增高的多行输入框。
- 输入框内的提交操作“开始编报”。
- 空输入校验和分析失败后的原地重试状态。

输入框接受事件主体、背景、事实、链接和关注方向的混合文本。提交后立即进入分析加载状态，不再要求用户点击“开始分析”。

### 3.2 第二阶段：事件分析

分析页使用对话式纵向阅读流：

- 用户原始输入以弱化背景的消息块显示。
- 系统分析以无边框正文区显示。
- 只展示事件概括、核心主体、时间与地点、关键事实、涉我风险。
- 页面底部保留“返回重新输入”和“生成提纲”。

分析中显示稳定高度的加载状态；失败时保留原始输入并提供重试。

### 3.3 第三阶段：拟稿提纲

提纲页改为宽版编辑工作区，不使用对话气泡承载复杂表单。

编辑内容包括：

- 建议标题。
- 主题立意。
- 核心判断。
- 一级目录。
- 二级目录。
- 各级章节说明。

用户进入此阶段后直接编辑，不需要先点击“编辑”。AI 修改入口位于编辑区下方，使用类似消息输入框的意见输入区。AI 修改前先提交尚未保存的手动修改，随后用当前最新提纲调用现有 AI 修改接口，并用返回结果覆盖编辑区。

页面固定展示保存状态：未保存、保存中、已自动保存、保存失败。保存失败时保留本地内容，显示重试操作，并阻止进入确认页或切换历史事件。

### 3.4 第四阶段：确认提纲

确认页只读展示标题、主题立意、核心判断、完整目录和章节说明。

不展示事件分析、AI 意见输入、编辑控件、版本信息或导入配置。操作只有“返回修改提纲”和“导入深度编报”。

### 3.5 第五阶段：导入深度编报

进入第五阶段后自动串联现有两步操作：

1. 调用提纲导入接口生成编报计划。
2. 调用创建报告任务接口创建开启深度编报的任务。

成功后保存当前前端会话中的任务信息，触发现有 `report-job-created` 事件并跳转原有深度编报页面。失败时保留事件和提纲，展示失败原因与“重新创建”操作。

## 4. 历史编报抽屉

历史记录参考 ChatGPT 的侧栏信息密度和交互，不复制其视觉品牌。

- 第一阶段完全不显示历史记录。
- 第二至第五阶段显示一个轻量历史入口。
- 抽屉默认收起，展开后覆盖主内容左侧，不挤压编辑区。
- 桌面宽度约 260px，移动端宽度不超过视口减 24px。
- 顶部提供“新建编报”和搜索。
- 列表按更新时间排序，使用直接列表而非卡片。
- 当前事件用浅灰背景和清晰文字突出。
- 切换事件前刷新当前自动保存队列；保存失败时停止切换。
- 搜索在前端对当前已加载事件的标题和摘要进行过滤。

现有接口只返回分页历史且没有删除接口，因此本次保留新建、搜索和切换，不显示无法完成的删除操作。

## 5. 前端状态与数据映射

前端阶段模型：

```ts
type DraftStage = 'input' | 'analysis' | 'outline' | 'confirm' | 'importing' | 'completed'
type SaveStatus = 'idle' | 'dirty' | 'saving' | 'saved' | 'error'
```

单一输入映射到现有分析接口：

- `title`：取第一条非空文本，清理列表符号后截断为合理标题长度。
- `materials`：保留完整原始输入。
- `links`：从完整输入中提取有效 HTTP/HTTPS URL。
- `category`、`region`：传空值。

历史事件恢复规则：

- 无提纲时恢复到分析阶段。
- 有最新提纲时恢复到提纲阶段。
- 确认和导入是当前浏览器会话状态；刷新后回到可继续操作的提纲阶段。

前端不再调用历史版本列表接口，也不再渲染版本时间线。`getDraftEvent` 返回的 `latestOutline` 和生成、保存、AI 修改接口的最新响应是唯一展示对象。

## 6. 自动保存

自动保存沿用 `manualUpdateDraftOutline`，采用防抖和串行队列：

1. 编辑字段变化后立即标记为 `dirty`。
2. 停止输入约 1.2 秒后发起保存。
3. 同一时刻只允许一个保存请求。
4. 保存期间产生的新修改在当前请求结束后继续保存。
5. 保存成功后用服务端返回的最新 `outlineId` 和提纲更新当前状态。
6. 保存失败时不覆盖本地编辑值，状态变为 `error`。
7. 进入确认、调用 AI 修改、切换历史和离开拟稿工作区前执行 `flush`。

由于后端保持现有行为，每次成功保存仍可能生成隐藏的历史版本。这是“仅前端改版”的已知兼容限制。

## 7. 组件边界

现有 `DraftAssistant.vue` 负责流程编排，逐步把展示职责拆出：

- `DraftSourceComposer`：单一输入和开始编报。
- `DraftAnalysisView`：五类只读分析。
- `DraftOutlineEditor`：提纲编辑和 AI 修改输入。
- `DraftOutlineConfirmation`：只读确认。
- `DraftImportState`：创建、失败和重试状态。
- `DraftHistoryDrawer`：历史搜索、列表和切换。

可复用现有提纲规范化、目录编辑和自动增高输入能力。旧的步骤导航、上下文面板、事件输入预览、版本时间线和版本相关计算从主流程移除。

## 8. 响应式与可访问性

- 主阅读区最大宽度约 840px，提纲编辑区可放宽到约 1040px。
- 小屏幕下历史抽屉覆盖全高内容区，提供明确关闭按钮和遮罩。
- 所有图标按钮提供可访问名称和悬停提示。
- 保存状态使用文字和图标共同表达，不只依赖颜色。
- 加载区域保持稳定尺寸，避免内容跳动。
- 键盘可聚焦历史入口、抽屉操作、提纲字段和主要流程按钮。

## 9. 错误处理

- 分析失败：保留原始输入，允许重试或返回编辑。
- 提纲生成失败：停留分析页，允许重试。
- AI 修改失败：保留当前编辑内容和意见文本。
- 自动保存失败：保留本地内容，阻止阶段切换，提供重试。
- 导入失败：停留导入状态，允许重新创建任务。
- 历史加载失败：不影响当前事件，抽屉内展示局部错误。

## 10. 测试与验收

前端测试覆盖：

- 单一输入到现有分析请求的映射。
- 首屏不包含步骤条、历史、分类、地区、完整度和预览。
- 分析提交后自动进入分析状态。
- 分析页只显示五类信息。
- 提纲页不包含任何版本文案或版本操作。
- 自动保存的 dirty、saving、saved、error 状态和串行行为。
- 保存失败时阻止确认和历史切换。
- 历史抽屉默认收起、搜索和事件切换。
- 确认页只读。
- 导入自动串联现有两个请求，并触发现有跳转事件。
- 桌面和移动视口下无重叠、横向溢出和文字遮挡。

完成后运行前端构建、相关单元测试，并使用浏览器在桌面和移动视口检查五个阶段。
