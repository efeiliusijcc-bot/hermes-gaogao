# 动态感知管理页简易操作化设计

## 背景

当前管理页直接展示“运行状态”“运行记录”“死信 Inbox”、`READY`、`PROCESSING`、`RETRY_PENDING`、`DEAD_LETTER`、运行 ID 和事件 ID。它准确反映了后台实现，却要求管理员理解消息队列和状态机，无法快速回答“今天有没有简报”“出了什么问题”“现在该点什么”。

本设计保留现有每日动态感知的生成、重试、死信和审计能力，只重组管理端的信息架构、显示文案和操作入口。后台状态仍然保留，技术详情默认收起。

## 目标与非目标

### 目标

- 管理员进入页面后，第一屏能判断当天简报是否完成。
- 失败时只展示可执行的问题和按钮，不要求理解 Inbox 或状态码。
- 历史记录按“每天一条最终结果”展示，重复重试收进详情。
- 配置只呈现类别和入选条数等业务设置；重试参数等技术设置折叠。
- 保留查看简报、重新生成、查看原因和展开技术详情的能力。

### 非目标

- 不改变 06:00 定时规则、前一天数据来源、模型评分和 50 条选取逻辑。
- 不删除 Inbox、运行记录、死信或审计数据。
- 第一阶段不新增数据库表，不改变现有内部事件契约；管理端接口保持兼容，如确有需要只允许增加向后兼容的只读汇总字段。

## 方案比较

### 方案一：只改字段名称

保留现有三个页签和表格，只将英文状态翻译成中文。

优点是改动最小；缺点是页面结构和后台思维方式不变，管理员仍要阅读运行 ID、批次和错误记录。放弃。

### 方案二：三页业务工作台（采用）

将入口改为“今日简报 / 异常处理 / 历史记录”。成功结果、待处理异常和历史查询分别服务一个明确任务；Inbox、运行和死信只作为内部数据来源。

优点是学习成本最低、操作路径短、可在现有接口上实现；缺点是需要一层前端状态映射，并要将少量当前简报数据补充到管理员首页。采用。

### 方案三：向导式处理中心

把“查看数据 → 生成 → 复核 → 发布”做成多步向导。

优点是流程解释最完整；缺点是与当前自动生成模式不匹配，会让管理员每次进入都面对不需要的步骤。暂不采用。

## 信息架构

管理页只保留三个一级页签：

1. **今日简报**：查看今天的最终结果和常用设置。
2. **异常处理**：只看仍需关注的问题。
3. **历史记录**：按天查看最终结果和必要的重新生成入口。

“生成设置”从今日简报页进入抽屉或弹窗，不再作为大块配置表单占据首屏。

## 页面一：今日简报

### 首屏内容

顶部结果区只显示业务语言：

- 主状态：`今日简报已生成`、`正在生成`、`等待数据`、`生成未完成` 或 `暂无可用数据`。
- 简报日期：当前业务日期。
- 数据日期：实际使用的前一天日期。
- 入选消息：实际简报事件数量；成功时由当前简报事件列表计算，不能只显示配置上限。
- 完成时间：成功简报的生成时间。

状态区旁保留一个“查看简报”按钮。生成中按钮显示为禁用状态；失败或暂无数据时，主按钮改为“查看处理办法”，跳转到异常处理页。

### 设置入口

“生成设置”显示一行摘要，例如“涉政、危安、涉华、其他 · 每天 50 条”。点击“修改设置”打开表单：

- 动态类别：固定类别复选框。
- 每天入选：数字输入，范围 1–50，默认 50。
- 生成时间：展示“每天 06:00”，只读说明，不提供管理员修改入口。
- 高级设置：折叠区域，包含最大重试次数、重试间隔、摘要最大长度；保留现有 API 字段，避免破坏配置契约。

保存时保留当前版本号，遇到版本冲突以“设置已被其他管理员更新，请刷新后再试”提示。

### 重新生成

“重新生成”放在次要操作区域，不与“查看简报”抢主视觉。点击后打开确认弹窗，要求填写原因并明确确认覆盖；如果当天已经有成功的全局简报，默认不允许 Inbox 重处理覆盖它，沿用现有幂等边界。

## 页面二：异常处理

### 展示原则

该页只展示尚未解决的问题。已解决的事件不在列表中长期占位；没有异常时显示“当前一切正常”。

列表字段：

- 简报日期。
- 发生了什么：用可读错误描述，不直接展示错误代码。
- 处理状态。
- 操作。

### 状态文案映射

| 后台状态 | 管理员看到的文案 | 操作 |
| --- | --- | --- |
| `PROCESSING` | 正在生成 | 只读，显示刷新 |
| `RETRY_PENDING` | 正在自动恢复 | 显示下次尝试时间，可查看原因 |
| `DEAD_LETTER` | 需要人工处理 | 显示“再次生成”和“查看原因” |
| `RECEIVED` | 即将开始 | 只读，显示刷新 |
| `PROCESSED` | 已解决 | 从异常列表移除，进入历史详情 |

“再次生成”仍调用现有重处理 API，并在服务端继续检查“仅死信、且不存在成功全局简报”的条件。操作成功后立即显示“已重新排队，系统正在处理”。

### 错误详情

默认只显示一句原因。展开后显示：发生时间、自动尝试次数、最后一次尝试时间、原始错误信息、事件 ID 和运行 ID。技术字段使用“技术详情”标题，不出现在默认表格中。

## 页面三：历史记录

### 列表

历史按业务日期降序，一天只占一行：

- 简报日期。
- 最终结果：`生成成功`、`补生成成功`、`未生成`、`暂无数据`。
- 数据来源：显示“使用 7 月 19 日数据”，不优先显示表名。
- 入选消息：实际数量。
- 完成时间。
- 操作：`查看简报`、`查看详情` 或 `重新生成`。

筛选只保留日期范围和结果，默认最近 30 天。历史行的详情抽屉展示所有运行尝试、技术 ID、来源表、错误代码和审计信息。

### 失败记录

`GENERATION_FAILED` 映射为“未生成”，并显示“重新生成”；`NO_DATA` 映射为“暂无数据”，只显示数据说明，不显示危险色或重试按钮，除非后端仍有可重试异常。

## 前端数据与接口边界

- 继续使用现有管理员状态、配置、运行、Inbox 列表、重处理和重新生成接口。
- 首页为显示实际入选数量，需要读取当前简报的 `displayedBrief.events.length`，或在管理员状态响应中补充等价的只读汇总字段；不读取数据库内部 ID 作为用户文案。
- 前端建立集中式状态映射函数，禁止在模板中散落 `DEAD_LETTER` 等技术枚举。
- 重新生成和再次处理按钮继续受 `system:daily-awareness:manage` 权限保护。
- 用户只读页面和下载、Word、Markdown 功能保持不变。

## 错误与边界

- 加载失败：页面显示“暂时无法获取动态感知状态”，保留“重试”按钮。
- 生成中：禁止重复点击生成操作，显示最近一次开始时间。
- 成功简报存在时：Inbox 重处理继续返回冲突，不隐式覆盖成功简报。
- 空列表：三个页面分别提供明确空状态，不显示空表头或 `--` 堆叠。
- 错误信息过长：默认截断，详情抽屉允许查看完整文本。
- 小屏：状态摘要两列排列，历史表格变为可横向滚动或详情行，不改变操作顺序。

## 验证方案

- 增加前端测试：各后台状态映射到正确业务文案和可用操作。
- 增加前端测试：成功首页显示实际事件数量，不能只显示配置上限。
- 增加前端测试：死信再次处理按钮仅在 `DEAD_LETTER` 出现，成功简报冲突时显示可读提示。
- 增加前端测试：历史记录按天合并展示，技术详情默认收起。
- 运行现有每日动态感知测试集和前端测试，执行服务端构建。
- 在桌面和窄屏浏览器检查三页的滚动、按钮禁用、空状态和错误提示。

## 设计结论

采用方案二。第一阶段聚焦 `DailyAwarenessAdmin.vue` 的信息架构、状态文案、操作按钮和设置抽屉；只有在现有接口无法提供实际入选数量时，才增加一个最小只读汇总字段，不改生成流程和数据表。
